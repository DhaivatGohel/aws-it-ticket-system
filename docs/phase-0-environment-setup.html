Why environment setup is not boring

Most tutorials skip straight to the interesting stuff — deploying Lambda, wiring up API Gateway. But in every real cloud engineering role, the first thing you do on a new project is establish secure, auditable access before touching a single service. Getting this wrong means leaked credentials, unconstrained blast radius if something goes wrong, and no paper trail.

Phase 0 is where you prove you think like an engineer, not just a builder.

Step 0.1 — IAM User and access keys

The principle

AWS root account credentials are the master key to your entire cloud environment. You never use root for day-to-day work. Instead, you create an IAM (Identity and Access Management) user — a scoped identity with its own credentials that can be rotated, audited, or revoked independently.

DECISION
Why attach permissions via a group, not directly to the user?

My IAM user dsgohel already existed with AdministratorAccess attached through a group called admin. This is better practice than attaching policies directly to the user.

In a corporate environment, you manage permissions at the group level so you can add or remove users from teams without having to re-configure individual policies. Even for a solo project, forming good habits matters — your future colleagues will thank you.

⚠

Known tradeoff: AdministratorAccess for a solo project

In a real team environment, you'd scope the IAM user's permissions to exactly the services it needs — S3, Lambda, DynamoDB, SES, etc. — and nothing more. That's called least-privilege. For this portfolio project, AdministratorAccess is acceptable because I'm the only user and the account is isolated to this build. I'll tighten IAM roles per-service as we go.

⚠

Security gap: Console access without MFA

The IAM console flagged that my user has console access enabled without multi-factor authentication. In any production corporate account, MFA on IAM users is mandatory — it's often enforced by an IAM policy that blocks console access unless MFA is active. This is worth fixing for real. Note it, improve it, and mention it — that's what a security-conscious engineer does.

Generating access keys

Access keys are how the AWS CLI authenticates on your behalf. They come in a pair: an Access Key ID (like a username) and a Secret Access Key (like a password). The secret is shown exactly once at creation — if you lose it, you have to rotate the key.

AWS now recommends SSO (aws sso login) or CloudShell as alternatives to long-term access keys. For an enterprise team managing dozens of engineers, SSO is the right call — it's centrally managed and time-scoped. For a solo dev project where I need CLI access from my own Windows machine and will be wiring GitHub Actions later, access keys are the pragmatic choice.

Step 0.2 — Configuring AWS CLI

aws configure

# AWS prompts you for each value interactively:
AWS Access Key ID [None]: AKIA...
AWS Secret Access Key [None]: ********
Default region name [None]: ca-central-1
Default output format [None]: json
DECISION
Why ca-central-1 (Canada Central — Montreal)?

This is not a default — it's a deliberate architectural decision. ca-central-1 is AWS's Montreal region, and choosing it matters for two reasons:

Data residency: Canadian privacy law (PIPEDA) requires that certain categories of personal data stay within Canada. Any IT ticketing system that handles employee information should be built in a Canadian region from day one. Retrofitting this later is painful.

Latency: I'm in Toronto. Montreal is ~500km away. US-East (Virginia) is ~900km. Lower latency, same price.

This is exactly the kind of region selection rationale you'd document in an Architecture Decision Record (ADR) at a company like Telus or Bell.

After configuring, I verified the credentials with:

aws sts get-caller-identity
{ "UserId": "AIDAZYUQOXF46HQWTWPSG", "Account": "[ACCOUNT-ID]", "Arn": "arn:aws:iam::[ACCOUNT-ID]:user/dsgohel" }

sts get-caller-identity calls AWS's Security Token Service and asks "who am I?" The response confirms the CLI is authenticated as user/dsgohel — not root, not some misconfigured profile. This is the first thing you run after any credential change.

Step 0.3 — GitHub repository

Every infrastructure change, every Lambda edit, every config tweak lives in version control. This is non-negotiable in cloud engineering. The repo also serves three other purposes in this project:

  • The CI/CD pipeline in Phase 4 watches this repo and auto-deploys on push

Step 0.4 — Folder structure

A consistent folder structure is professional discipline. When a senior engineer or hiring manager opens your repo cold, they should navigate it without asking you anything. 

aws-it-ticket-system/
├── frontend/          # S3-hosted HTML/CSS/JS (web form + admin dashboard)
├── lambda/            # Python Lambda function code
├── infrastructure/    # IAM policies, CloudFormation if needed
├── docs/              # Architecture diagrams, ADRs, notes
├── .gitignore         # Python gitignore (excludes __pycache__, .pyc, etc.)
├── LICENSE            # MIT
└── README.md

Each folder has a .gitkeep file — a blank placeholder. Git doesn't track empty directories, so this is the standard convention to commit a folder that will be populated later. 

# Commit message follows Conventional Commits spec
git commit -m "chore: initial project structure"
Engineering insight

The Conventional Commits spec (feat:, fix:, chore:, docs:) is used by most modern engineering teams. It enables automated changelog generation and makes the commit history readable to anyone on the team. Starting it from the first commit costs nothing and signals that you take your craft seriously.
