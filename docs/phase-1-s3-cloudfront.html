The architecture

This phase establishes the static hosting layer — the public-facing entry point of the entire system. Two services, one pattern used by virtually every production static site on AWS.

The key word in that diagram is private. The S3 bucket is completely locked down — no public access whatsoever. CloudFront is the only entity allowed to read from it, enforced by a bucket policy scoped to the exact distribution ID. This is called Origin Access Control (OAC).

Why not just use S3 static website hosting?

S3 has a built-in static website hosting feature. It's tempting — one checkbox and you have a URL. But it has two serious problems for a real deployment:

S3 static hosting alone

HTTP only — no HTTPS. Browsers flag it as insecure. No caching. Every request hits S3 in Montreal. No access control at the edge.

S3 + CloudFront

HTTPS by default. Files cached at 400+ edge locations. Bucket stays private. Security headers, WAF, and geo-restriction all available at the edge.

In 2026, serving a site over HTTP is not acceptable — browsers show security warnings, and search engines penalize it. CloudFront gives you HTTPS with AWS's own certificate at no extra cost.

Step 1.1 — Creating the S3 bucket

The command

aws s3api create-bucket \
  --bucket aws-it-ticket-system-dsgohel \
  --region ca-central-1 \
  --create-bucket-configuration LocationConstraint=ca-central-1
DECISION
Why does the bucket name include my username?

S3 bucket names are globally unique across every AWS account in the world. If you create a bucket called my-website, no one else on the planet can use that name. Adding a personal identifier like dsgohel avoids collisions and is standard practice in real environments where bucket names often include the team name, environment, and region.

⚠

Why the LocationConstraint flag?

Every AWS region except us-east-1 requires the --create-bucket-configuration LocationConstraint flag. This is a legacy quirk — S3 was originally built in us-east-1 and region support was added later. Omitting this flag silently creates your bucket in Virginia, not Montreal. Always include it for non-us-east-1 deployments.

Step 1.2 — Blocking all public access

aws s3api put-public-access-block \
  --bucket aws-it-ticket-system-dsgohel \
  --public-access-block-configuration \
  "BlockPublicAcls=true,IgnorePublicAcls=true,BlockPublicPolicy=true,RestrictPublicBuckets=true"

All four flags set to true. This is the maximum lockdown — no ACLs, no public policies, no exceptions. The bucket is now invisible to the public internet. The only way anything gets served is through CloudFront, which we control.

Security mindset

This feels counterintuitive — you're building a public website but locking down the bucket. The distinction is: the website is public, the storage is not. CloudFront is the public interface. S3 is the private warehouse. This separation is what lets you add access controls, rate limiting, and security headers at the edge without touching your files.

Step 1.3 — CloudFront with Origin Access Control

What is OAC?

Origin Access Control is the mechanism that lets CloudFront authenticate itself to S3. When CloudFront fetches a file from your private bucket, it signs the request with AWS SigV4 — the same signing protocol used for IAM API calls. S3 verifies the signature and only serves the file if it comes from the authorized distribution.

ℹ

OAC vs OAI — why it matters for your resume

OAI (Origin Access Identity) was the previous way to do this — deprecated by AWS in 2022. Most tutorials still use OAI. Using OAC signals that you're following current AWS documentation, not copying outdated blog posts. Interviewers who know AWS will notice.

The distribution config

{
  "CallerReference": "aws-it-ticket-system-001",
  "DefaultRootObject": "index.html",
  "ViewerProtocolPolicy": "redirect-to-https",  // HTTP → HTTPS always
  "CachePolicyId": "658327ea-...",            // AWS managed CachingOptimized
  "PriceClass": "PriceClass_100",              // NA + Europe only — lower cost
  "Compress": true,                           // auto gzip compression
  "HttpVersion": "http2"                        // multiplexed connections
}
DECISION
PriceClass_100 — why limit edge locations?

CloudFront has three price classes. PriceClass_All uses every edge location globally including Asia-Pacific and South America. PriceClass_100 covers North America and Europe only.

My users are in Canada. There's no reason to pay for edge nodes in Singapore or São Paulo. PriceClass_100 gives full coverage for the relevant geography at lower cost — exactly the kind of cost-aware architecture decision you'd make at a Canadian company.

Step 1.4 — Bucket policy scoped to the distribution

{
  "Effect": "Allow",
  "Principal": { "Service": "cloudfront.amazonaws.com" },
  "Action": "s3:GetObject",
  "Resource": "arn:aws:s3:::aws-it-ticket-system-dsgohel/*",
  "Condition": {
    "StringEquals": {
      "AWS:SourceArn": "arn:aws:cloudfront::[ACCOUNT-ID]:distribution/[DISTRIBUTION-ID]"
    }
  }
}

This policy is the enforcement layer. It permits s3:GetObject — reading files — but only when the request comes from distribution [DISTRIBUTION-ID] in account [ACCOUNT-ID]. Not from any other CloudFront distribution. Not from another AWS account. Not from the public. Exactly one entity, exactly one permission. This is least-privilege applied correctly.

Verifying the deployment

aws cloudfront get-distribution \
  --id [DISTRIBUTION-ID] \
  --query "Distribution.Status"
"Deployed"

CloudFront distributions take 5–15 minutes to propagate across edge locations after creation. The status moves from InProgress to Deployed once all edge nodes have the configuration. After that, the site is live globally.

A note on the Windows encoding issue

One thing worth documenting: PowerShell's Set-Content writes files with a UTF-8 BOM (Byte Order Mark — the ï»¿ prefix) by default. The AWS CLI cannot parse JSON files with a BOM, throwing a cryptic Expected: '=' error. The fix is to always write JSON config files in VS Code with UTF-8 encoding (not UTF-8 with BOM), or use Git Bash for AWS CLI work on Windows. Git Bash behaves like a Linux terminal and avoids the encoding mismatch entirely.

Documenting failures is part of good engineering writing. Anyone hitting this issue will find the answer here.
