The problem with manual deployments

Before Phase 4, deploying any change to this project required running 4–5 commands manually every single time:

Before — manual deployment
  • Edit code locally
  • Open PowerShell, zip Lambda manually
  • Run aws lambda update-function-code
  • Run aws s3 cp for each changed file
  • Run aws cloudfront create-invalidation
  • Hope you didn't forget a step
After — automated pipeline
  • Edit code locally
  • git push origin main
  • Done — everything else is automatic

Manual deployments are slow, error-prone, and don't scale. In a real engineering team, nobody deploys by hand — every company from a 5-person startup to a bank uses a CI/CD pipeline. Building one from scratch is a core DevOps skill.

Plain English

CI/CD is like a factory assembly line. Before it existed, every car was hand-built from scratch by one person doing every step. The assembly line breaks it into automated steps that run in sequence — faster, more consistent, and less error-prone. GitHub Actions is your assembly line for code deployments.

What CI/CD actually means

CI — Continuous Integration: Every time you push code, it's automatically tested and validated. Catches bugs before they reach production.

CD — Continuous Deployment: Every time you push to main, the code is automatically deployed to production. No manual steps, no human error.

For this project we're focused on CD — automatic deployment on push. The pipeline deploys the frontend to S3, invalidates the CloudFront cache, and redeploys the Lambda function every time code changes.

Step 4.1 — Dedicated IAM user for GitHub Actions

GitHub Actions needs AWS credentials to deploy on your behalf. The wrong approach would be using your personal dsgohel credentials — that user has full AdministratorAccess to your entire AWS account. If those credentials leaked through GitHub, an attacker could do anything.

Instead, a dedicated github-actions-deploy IAM user was created with exactly three permissions:

  • S3 — PutObject, GetObject, DeleteObject, ListBucket on the ticket system bucket only
  • CloudFront — CreateInvalidation on this specific distribution only
  • Lambda — UpdateFunctionCode on this specific function only

Nothing else. Not IAM management, not DynamoDB, not SES, not any other S3 bucket. This is least-privilege applied to CI/CD — a pattern every security-conscious engineering team uses.

A security incident — and why it matters

Security incident — documented honestly

During setup, the aws iam create-access-key output was accidentally pasted into this build log — exposing the AccessKeyId and SecretAccessKey in plain text.

The response was immediate: the exposed keys were deleted within minutes using aws iam delete-access-key, and fresh credentials were generated before proceeding.

This is the correct response to a credential exposure. AWS access keys starting with AKIA are actively scanned for by bots on GitHub, chat logs, and public forums. Exposed credentials have been used to spin up hundreds of EC2 instances for crypto mining within minutes — resulting in bills of thousands of dollars.

The lesson: never paste credentials into any interface that isn't a secrets manager. If it happens — rotate immediately, no exceptions. Document it, learn from it, move on.

Step 4.2 — GitHub Secrets

GitHub Secrets are encrypted key-value pairs stored separately from your code. Your workflow file references them by name — the actual values are never visible in the repository, never in logs, never anywhere public.

Six secrets were configured for this pipeline:

AWS_ACCESS_KEY_ID          # github-actions-deploy credentials
AWS_SECRET_ACCESS_KEY      # github-actions-deploy credentials
AWS_REGION                 # ca-central-1
S3_BUCKET                  # aws-it-ticket-system-dsgohel
CLOUDFRONT_DISTRIBUTION_ID # the CloudFront distribution ID
LAMBDA_FUNCTION_NAME       # it-ticket-triage
DECISION
Why secrets for non-sensitive values like bucket name?

The S3 bucket name and Lambda function name aren't credentials — they're not sensitive in the same way. But storing them as secrets means the workflow file contains zero hardcoded values. If you ever rename a resource, you update one secret in GitHub — not the workflow file. It also means the workflow file is completely portable and reusable across environments.

Step 4.3 — The workflow file

The workflow file lives at .github/workflows/deploy.yml. GitHub automatically detects and runs any YAML files in that folder. Here's the complete file with every line explained:

# The name shown in the GitHub Actions UI
name: Deploy to AWS

# Trigger: run this workflow on every push to main
on:
  push:
    branches:
      - main

jobs:
  deploy:
    # GitHub spins up a fresh Ubuntu Linux VM for each run
    runs-on: ubuntu-latest

    steps:
      # Step 1 — download your repo onto the Linux VM
      - name: Checkout code
        uses: actions/checkout@v4

      # Step 2 — install AWS CLI and inject your credentials
      - name: Configure AWS credentials
        uses: aws-actions/configure-aws-credentials@v4
        with:
          aws-access-key-id:     ${{ secrets.AWS_ACCESS_KEY_ID }}
          aws-secret-access-key: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
          aws-region:            ${{ secrets.AWS_REGION }}

      # Step 3 — sync frontend/ folder to S3
      # --delete removes files in S3 that no longer exist locally
      - name: Deploy frontend to S3
        run: |
          aws s3 sync frontend/ s3://${{ secrets.S3_BUCKET }} \
            --delete \
            --exclude "*.DS_Store"

      # Step 4 — tell CloudFront to serve fresh files immediately
      - name: Invalidate CloudFront cache
        run: |
          aws cloudfront create-invalidation \
            --distribution-id ${{ secrets.CLOUDFRONT_DISTRIBUTION_ID }} \
            --paths "/*"

      # Step 5 — zip and redeploy the Lambda function
      - name: Deploy Lambda function
        run: |
          cd lambda
          zip triage_function.zip triage_function.py
          aws lambda update-function-code \
            --function-name ${{ secrets.LAMBDA_FUNCTION_NAME }} \
            --zip-file fileb://triage_function.zip \
            --region ${{ secrets.AWS_REGION }}

1
Checkout code
GitHub spins up a fresh Ubuntu VM and downloads your repo onto it

Engineering insight

The pipeline always deploys both frontend and Lambda — even if only one changed. This is simpler and more reliable than trying to detect which files changed and running conditional steps. At this scale, a few extra seconds of deployment time is worth the reduction in complexity. In a larger project with longer build times, you'd add path filters to only deploy what changed.
