Why serverless?

A traditional server approach means provisioning an EC2 instance — running 24/7, billing whether it's handling requests or sitting idle. For a ticketing system that might get 100 submissions a day, you're paying for idle compute 99% of the time.

With Lambda, you pay only when a ticket is actually submitted. The function wakes up, runs for under a second, and goes back to sleep. No server to patch, no OS to maintain, no capacity to provision. At portfolio traffic levels, the entire backend costs fractions of a cent per month.

Plain English

Lambda is like a contractor who only shows up when there's work to do. A traditional server is an employee you pay full-time even when there's nothing to do. For unpredictable, low-volume workloads, the contractor model wins on cost and simplicity.

DynamoDB: the filing cabinet

Every ticket needs to be stored permanently. DynamoDB is AWS's NoSQL database — a flexible filing cabinet where each item (ticket) can have different fields.

DECISION
Why DynamoDB over RDS?

IT tickets are naturally document-shaped. A Hardware ticket might include a device serial number. An Access ticket might include the system being locked. A rigid SQL schema would require adding columns for every possible field type — awkward and inflexible.

DynamoDB's schema-free model handles this cleanly. Each ticket is one item with whatever fields make sense for that type. DynamoDB also costs nothing at idle with PAY_PER_REQUEST billing — no minimum instance cost like RDS requires.

The table has one required field — ticketId — a unique string identifier. Every other field (name, email, description, category, priority, status, timestamp) is added freely at write time. That flexibility is why NoSQL suits this use case.

IAM: the key card system

Lambda needed permissions to write to DynamoDB and emit logs to CloudWatch. The correct way to grant these is through an IAM execution role — not hardcoded credentials, not environment variables with access keys. Never that.

Plain English

IAM is like a key card system in an office building. The janitor's card opens the supply room but not the server room. Lambda gets exactly the key cards it needs — nothing more. Nobody gets a master key.

✕

The most common Lambda security mistake

Hardcoding AWS access keys inside Lambda function code. Those keys get committed to GitHub, scraped by bots within minutes, and result in compromised accounts and surprise bills. IAM roles give Lambda temporary, automatically rotating credentials — no static keys ever needed.

The Lambda role received exactly two permissions — no more:

  • AWSLambdaBasicExecutionRole — write logs to CloudWatch. Without this, debugging is impossible.
  • Custom DynamoDB policy — four actions on exactly one table: PutItem, GetItem, Scan, Query. No delete, no update, no access to other tables.

Lambda: the brain

The Lambda function receives a ticket, figures out what kind of problem it is, how urgent it is, and saves it — all in under 500 milliseconds.

Categorization — keyword scanning

No AI involved. The function lowercases the description and checks for keywords:

def categorize(description):
    text = description.lower()  # "Laptop" and "laptop" both match

    if any(word in text for word in ['laptop', 'printer', 'monitor']):
        return 'Hardware'
    elif any(word in text for word in ['vpn', 'wifi', 'network']):
        return 'Network'
    elif any(word in text for word in ['password', 'login', 'locked']):
        return 'Access'
    elif any(word in text for word in ['app', 'crash', 'install', 'error']):
        return 'Software'
    else:
        return 'General'  # no keywords matched

Prioritization — urgency first, category as tiebreaker

def prioritize(description, category):
    text = description.lower()

    if any(word in text for word in ['urgent', 'outage', 'breach', 'down']):
        return 'Critical'
    elif any(word in text for word in ['broken', 'not working', 'cant access']):
        return 'High'
    elif category in ['Network', 'Access']:
        return 'Medium'  # connectivity issues default to medium
    else:
        return 'Low'

Test result — before touching the frontend

{"message": "Ticket submitted successfully", "ticketId": "b4b582a6-...", "category": "Hardware", "priority": "High"}

"laptop" → Hardware. "cant access" → High. Both correct. This was verified with aws lambda invoke directly — before API Gateway was even connected.

Engineering insight

Always test Lambda directly before building the frontend. If the form doesn't work later, you know the problem is in the UI — not somewhere in the backend stack. Isolating layers is how you debug efficiently.

API Gateway: the receptionist

Lambda can't be called directly from a browser — it needs a public HTTPS endpoint. API Gateway creates that endpoint and handles routing, TLS termination, and request routing.

Plain English

API Gateway is like a receptionist at the front desk. It receives requests from the outside world, checks they're valid, and directs them to the right person (Lambda). It doesn't do the actual work — it manages traffic.

Three methods were configured on the /tickets resource: POST (submit ticket), GET (list all tickets for dashboard), and OPTIONS (browser preflight — explained next).

The CORS battle — documented honestly

This is the part most tutorials skip. Every AWS developer hits CORS eventually. Here's exactly what happened and how it was fixed.

What is CORS?

Your website lives at d23mh63awvqncl.cloudfront.net. Your API lives at a different address. Browsers have a built-in security rule: if a website tries to talk to a different address, ask for permission first.

This "asking for permission" is called a preflight request. The browser sends an OPTIONS request saying: "Hey API, is it okay if this website sends data to you?" If the API responds with the correct permission headers, the browser proceeds. If not, it blocks the request entirely — and shows a confusing error.

The error

Access to fetch at '[API-URL]' from origin 'https://d23mh63awvqncl.cloudfront.net' has been blocked by CORS policy: No 'Access-Control-Allow-Origin' header is present on the requested resource.

The diagnosis

Testing the OPTIONS preflight directly with curl — bypassing the browser entirely — revealed the root cause:

curl -X OPTIONS [API-URL] → HTTP/1.1 500 Internal Server Error

The OPTIONS preflight was returning a 500. API Gateway couldn't process it at all. This is an API Gateway configuration problem — not a frontend issue, not a Lambda issue.

The fix

The correct pattern for CORS with API Gateway proxy integration requires two different integration types:

  • POST and GET → AWS_PROXY to Lambda. Lambda returns CORS headers in its response body.
  • OPTIONS → MOCK integration. API Gateway responds immediately with CORS headers — without calling Lambda at all.

The MOCK integration needs three components in the correct order: a request template telling it to return 200, a method response declaring which headers are allowed, and an integration response setting the actual header values.

Verification

HTTP/1.1 200 OK Access-Control-Allow-Methods: GET,POST,OPTIONS Access-Control-Allow-Origin: * Access-Control-Allow-Headers: Content-Type
⚠

Lesson learned

Always verify CORS at the API level with curl before debugging the frontend. The browser error message points at the symptom, not the cause. A 500 on the OPTIONS preflight is always an API Gateway misconfiguration — diagnose at the API layer first.
