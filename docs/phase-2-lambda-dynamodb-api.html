Why serverless for an IT ticketing system?

The traditional approach would be an EC2 instance running a web server — always on, always billing. For a ticketing system that might receive 50 submissions a day, you'd be paying for idle compute 99% of the time.

With Lambda, you pay only when a ticket is actually submitted. At portfolio traffic levels, the Lambda and API Gateway costs are effectively zero — fractions of a cent per month. More importantly, there's no server to patch, no OS to maintain, no capacity to provision. This is exactly the cost and operational efficiency argument you'd make when proposing a serverless architecture at a Canadian company.

Step 2.1 — DynamoDB table

aws dynamodb create-table \
  --table-name it-tickets \
  --attribute-definitions AttributeName=ticketId,AttributeType=S \
  --key-schema AttributeName=ticketId,KeyType=HASH \
  --billing-mode PAY_PER_REQUEST \
  --region ca-central-1
DECISION
Why DynamoDB and not RDS?

IT tickets are naturally document-shaped — each ticket has a different set of fields depending on its type. A Hardware ticket might include a device serial number. An Access ticket might include the system being accessed. DynamoDB's schema-free model handles this cleanly. RDS would require either a rigid schema or a JSONB column hack.

DynamoDB also scales to zero cost at idle with PAY_PER_REQUEST billing — no minimum instance cost like RDS. For a system with unpredictable, low traffic this is the correct choice.

DECISION
Why PAY_PER_REQUEST billing mode?

DynamoDB has two billing modes. Provisioned capacity requires you to predict read/write throughput upfront and pay for it whether you use it or not. PAY_PER_REQUEST charges per operation — ideal for unpredictable or low-volume workloads. At portfolio scale this costs less than $0.01/month. In a high-volume production system with predictable traffic, provisioned capacity would be cheaper.

Step 2.2 — IAM role for Lambda

Lambda needs permissions to write to DynamoDB and to emit logs to CloudWatch. The correct way to grant these is through an IAM execution role — not hardcoded credentials, not environment variables with access keys. Never that.

✕

The most common Lambda security mistake

Hardcoding AWS credentials inside Lambda function code. This gets committed to GitHub, scraped by bots within minutes, and results in compromised accounts and unexpected bills. IAM roles exist precisely to eliminate this pattern. Lambda automatically receives temporary, rotating credentials through its execution role — no keys required.

The role was created with two policies attached:

  • AWSLambdaBasicExecutionRole — managed policy that allows writing logs to CloudWatch. Without this, debugging is impossible.
  • Custom DynamoDB policy — scoped to exactly four actions on exactly one table: PutItem, GetItem, Scan, Query. No DeleteItem. No UpdateItem. No access to any other table.
{
  "Effect": "Allow",
  "Action": [
    "dynamodb:PutItem",    // write new ticket
    "dynamodb:GetItem",    // fetch one ticket by ID
    "dynamodb:Scan",       // list all tickets (admin dashboard)
    "dynamodb:Query"       // filter tickets by criteria
  ],
  "Resource": "arn:aws:dynamodb:ca-central-1:[ACCOUNT-ID]:table/it-tickets"
}

Step 2.3 — The triage Lambda function

The Lambda function is the core of the system. It receives a ticket submission, categorizes it, assigns a priority, and persists it to DynamoDB — all in under 500ms.

Categorization logic

The function scans the ticket description for keywords and assigns one of five categories:

Hardware

laptop, computer, printer, monitor, keyboard, mouse

Network

wifi, internet, vpn, connection, ethernet

Access

password, login, account, locked, permission

Software

app, install, crash, error, update, virus

Priority logic

Priority is assigned based on urgency keywords in the description, with category as a tiebreaker:

def prioritize(description, category):
    text = description.lower()

    if any(word in text for word in ['urgent', 'critical', 'down', 'outage', 'breach']):
        return 'Critical'
    elif any(word in text for word in ['slow', 'not working', 'broken', 'failed']):
        return 'High'
    elif category in ['Network', 'Access']:
        return 'Medium'   # connectivity issues default to medium
    else:
        return 'Low'

The ticket object written to DynamoDB

{
  "ticketId":    "b4b582a6-b71f-4945-bb09-96dc22dde6ac",  // uuid4()
  "name":        "John Smith",
  "email":       "john@example.com",
  "description": "My laptop is broken and I cant access the network",
  "category":    "Hardware",
  "priority":    "High",
  "status":      "Open",
  "createdAt":   "2026-04-01T22:28:35.000352+00:00"
}
Engineering insight

Why store category and priority in DynamoDB rather than computing them on read? Because DynamoDB reads should be fast and cheap. Doing keyword scanning on every dashboard load would add latency and complexity. Computing on write and storing the result is the correct pattern — it's called write-time enrichment and is standard in event-driven architectures.

Step 2.4 — API Gateway

API Gateway creates the public HTTPS endpoint that the web form POSTs to. It sits between the browser and Lambda, handling TLS termination, request routing, and throttling.

DECISION
Why AWS_PROXY integration?

API Gateway supports two integration modes. Non-proxy mode lets you transform requests and responses using mapping templates — powerful but complex. AWS_PROXY passes the entire request to Lambda as-is and returns Lambda's response directly to the client. For a new API where Lambda already handles all the logic, proxy integration is simpler, more maintainable, and what AWS recommends.

⚠

Why OPTIONS method matters

Browsers send a CORS preflight OPTIONS request before any cross-origin POST. If API Gateway doesn't accept OPTIONS, the browser silently blocks the actual request — and you get a confusing error with no obvious cause. Both POST and OPTIONS must be configured and connected to Lambda. Our Lambda function handles the OPTIONS response directly, returning the correct CORS headers.

Cross-service permissions

Even after wiring API Gateway to Lambda in the console, Lambda still needs to explicitly trust API Gateway to invoke it. This is AWS's bilateral permission model — both sides must consent. The permission is scoped tightly: 

"Action": "lambda:InvokeFunction",
"Principal": "apigateway.amazonaws.com",
"Condition": {
  "ArnLike": {
    "AWS:SourceArn": "arn:aws:execute-api:ca-central-1:[ACCOUNT-ID]:[API-ID]/*/POST/tickets"
  }
}

Only this specific API, this specific method, this specific resource. Not any API Gateway in the account.

End-to-end test results

Before building the frontend, I validated the entire pipeline with a direct curl call:

curl -X POST \
  https://[API-ID].execute-api.ca-central-1.amazonaws.com/prod/tickets \
  -H "Content-Type: application/json" \
  -d '{"name": "Jane Doe", "email": "jane@example.com", "description": "Cannot login to VPN, account seems locked"}'
{"message": "Ticket submitted successfully", "ticketId": "5730022e-2081-4924-92f0-059162102ccd", "category": "Network", "priority": "Medium"}

VPN → `Network` category. Network category → `Medium` priority. Ticket written to DynamoDB and retrievable by ID. The full pipeline working over the public internet via HTTPS in under 500ms.

✦

Always test before building the frontend. Validating the API with curl first means that when the form doesn't work, you know the problem is in the frontend — not somewhere in the backend stack. Isolating layers is how you debug efficiently.
