Why alerting matters

After Phase 2, the system could receive tickets and display them on a dashboard. But the IT team would have to remember to check the dashboard — which means tickets could sit unnoticed for hours. In a real IT environment, that's unacceptable for a Critical or High priority issue.

Phase 3 makes the system proactive. The moment a ticket is submitted, an email goes out automatically. Nobody needs to remember to check anything.

Plain English

The difference between Phase 2 and Phase 3 is like the difference between a suggestion box and a pager. Both collect requests — but only one actually gets someone's attention immediately.

Step 3.1 — SES (Simple Email Service)

SES is AWS's email sending service. Before it trusts you to send mail, it requires you to verify ownership of the email address. This is sandbox mode — a security measure that prevents abuse. In sandbox mode, you can only send to addresses you've explicitly verified.

DECISION
Why stay in SES sandbox mode for this project?

Moving out of sandbox requires submitting a request to AWS explaining your use case and expected send volume. For a portfolio project with one verified address, sandbox mode is perfectly sufficient — and it means we never accidentally spam anyone.

In a real corporate deployment, you'd request production access and verify your company domain so emails come from something like it-alerts@company.com rather than via amazonses.com.

The email alert format

Every ticket submission triggers this email to the IT team:

[HIGH] New IT Ticket — Access issue from Naman Vora
From: [it-admin@company.com] via amazonses.com To: [it-admin@company.com]
New IT support ticket submitted. Ticket ID: 54748f75-ab90-430d-8199-4784a3ada265 Submitted: 2026-04-07T00:15:21.163356+00:00 From: Naman Vora Email: [it-admin@company.com] Category: Access Priority: High Status: Open Description: Intune not working. --- View all tickets: https://d23mh63awvqncl.cloudfront.net/dashboard.html AWS IT Ticket Triage System — ca-central-1

The subject line format — [HIGH] New IT Ticket — Access issue from Naman Vora — is designed for inbox scanning. An IT engineer on call can read the priority and category without even opening the email.

Why email failure doesn't block ticket submission

The send_alert_email() function is wrapped in its own try/except block inside Lambda:

try:
    ses.send_email(...)
    print(f"Alert email sent for ticket {ticket['ticketId']}")
except Exception as e:
    # Email failure should not block ticket submission
    print(f"Email send failed: {str(e)}")

If SES has an outage, if the email address is temporarily unreachable, if anything goes wrong — the ticket still saves to DynamoDB and the user still gets a success response. This is called graceful degradation — a core principle of resilient system design. The primary function (storing the ticket) never fails because a secondary function (sending an alert) had a problem.

Engineering insight

DynamoDB write happens before the email call. This ordering is deliberate. If we sent the email first and then DynamoDB failed, the IT team would receive an alert for a ticket that doesn't exist in the system. Always persist data before triggering downstream side effects.

Step 3.2 — Expanded triage keywords

The original keyword lists covered basic IT scenarios. After testing with real ticket descriptions, several gaps were obvious — "Intune not working", "Teams keeps crashing", "phishing email" — none of these matched correctly. The keyword lists were expanded to cover real enterprise IT vocabulary:

Hardware

laptop, computer, printer, monitor, keyboard, mouse, hardware, screen, dock

Network

wifi, internet, vpn, connection, ethernet, network, bandwidth, ping

Access

password, login, account, locked, permission, intune, mfa, authenticator, azure, entra

Software

app, crash, install, error, update, virus, teams, outlook, office, microsoft, onedrive, sharepoint

Security

phishing, spam, suspicious, malware, ransomware, hacked, breach

DECISION
Why a new Security category routes to Medium priority?

Security incidents like phishing reports warrant immediate attention but aren't always active breaches. Routing them to Medium by default ensures they appear prominently on the dashboard without triggering the same urgency as a confirmed breach or full outage. If the description contains words like "ransomware" or "hacked", the priority function escalates to Critical regardless of category.

Step 3.3 — Billing alarm

This is the operational safety net that every AWS project should have from day one. A CloudWatch alarm monitors the EstimatedCharges metric — which AWS publishes once per day — and sends an SNS notification if the monthly bill exceeds the threshold.

Plain English

Think of it like a low balance alert on your bank account. You set a threshold — $2 USD in this case — and if your AWS bill crosses it at any point during the month, you get an email immediately. No surprises at month end.

Why SNS instead of emailing directly from CloudWatch?

CloudWatch alarms can't send emails directly. They publish to an SNS topic (Simple Notification Service) — a message bus that can fan out to multiple subscribers. One alarm can notify an entire on-call team by email, SMS, or Slack simultaneously. For this project we have one subscriber, but the pattern scales.

DECISION
Why us-east-1 for the billing alarm?

AWS billing metrics (EstimatedCharges) are a global service that AWS centralizes in us-east-1 — N. Virginia. Even though all our application resources live in ca-central-1, billing data is only published to us-east-1. This is one of the few cases where you genuinely have no choice about the region. The SNS topic and CloudWatch alarm must both be in us-east-1 to see the billing metric.

aws cloudwatch put-metric-alarm \
  --alarm-name          "aws-monthly-bill-over-2-dollars" \
  --metric-name         EstimatedCharges \
  --namespace           AWS/Billing \
  --statistic           Maximum \
  --period              86400 \         # check once per day
  --threshold           2 \            # $2 USD
  --comparison-operator GreaterThanThreshold \
  --evaluation-periods  1 \
  --treat-missing-data  notBreaching \ # no false alarms on missing data
  --region              us-east-1

The treat-missing-data notBreaching flag is important — AWS doesn't publish the billing metric every hour, only once per day. Without this flag, CloudWatch would treat the missing hourly data points as a potential problem and could trigger false alarms.

⚠

Enable billing alerts in your AWS account settings

CloudWatch billing alarms only work if billing alerts are enabled in your account. Go to AWS Console → Billing → Billing preferences → check "Receive CloudWatch Billing Alerts." Without this, the alarm exists but never receives data to evaluate.
